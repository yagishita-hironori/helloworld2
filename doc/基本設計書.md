# 基本設計書

**システム名:** Hello World Web Application
**作成日:** 2026-03-17
**対象リポジトリ:** https://github.com/yagishita-hironori/helloworld

---

## 1. システム概要

React SPA をフロントエンド、ASP.NET Core をバックエンド API として、AWS ECS/Fargate 上で稼働する Web アプリケーションのサンプル構成。ALB によるルーティング、RDS (SQL Server) によるデータ永続化、GitHub Actions による CI/CD を備える。

---

## 2. システムアーキテクチャ

```
[ ブラウザ ]
     │ HTTP
     ▼
   ALB (Application Load Balancer)
     ├─ /           → frontend コンテナ (React SPA / Nginx)
     └─ /api/*      → backend コンテナ (ASP.NET Core API)
                                            ↓
                                    RDS (SQL Server)
```

### 構成要素

| コンポーネント | 技術スタック | ホスティング |
|---|---|---|
| フロントエンド | React 18 / Vite 5.4 / Nginx 1.27 | ECS Fargate |
| バックエンド API | ASP.NET Core (.NET 8.0) | ECS Fargate |
| データベース | SQL Server (RDS) | Amazon RDS |
| ロードバランサー | Application Load Balancer | AWS ALB |
| コンテナレジストリ | Amazon ECR | AWS ECR |
| ログ管理 | Amazon CloudWatch Logs | AWS CloudWatch |
| シークレット管理 | AWS Secrets Manager | AWS Secrets Manager |

## 3. フロントエンド設計

### 3.1 技術仕様

| 項目 | 内容 |
|---|---|
| フレームワーク | React 18.3.1 |
| ビルドツール | Vite 5.4 |
| Web サーバー | Nginx 1.27 (Alpine) |
| コンテナポート | 80 |
| ローカル開発ポート | 5173 |

### 3.2 コンポーネント構成

```
frontend/
├─ index.html          # エントリーポイント HTML (<script type="module"> でJSXをロード)
├─ vite.config.js      # Vite 設定 (プロキシ設定含む)
├─ package.json
└─ src/
   ├─ index.jsx        # React ルートのマウント
   └─ App.jsx          # メインコンポーネント、/api/hello を呼び出して結果を表示
```

### 3.3 API 連携

- `App.jsx` が画面表示時に `fetch('/api/hello')` を呼び出す
- 取得した `message` フィールドを画面に表示する
- 取得失敗時は「API に接続できませんでした」を表示する

### 3.4 ローカル開発時のプロキシ設定

`vite.config.js` にプロキシを設定し、`/api/*` リクエストをバックエンドに転送する。

```js
server: {
  proxy: {
    '/api': 'http://localhost:5000',
  },
},
```

### 3.5 Nginx ルーティング設定

| パス | 動作 |
|---|---|
| `/` | SPA フォールバック (`try_files $uri /index.html`) |
| `/health` | `200 ok` を返す (ALB ヘルスチェック用) |

### 3.6 Docker ビルド (マルチステージ)

| ステージ | ベースイメージ | 処理 |
|---|---|---|
| build | node:20-alpine | `npm ci --include=dev` → `npm run build` (`dist/` に出力) |
| runtime | nginx:1.27-alpine | `dist/` と `nginx.conf` をコピー |

## 6. CI/CD 設計

### 6.1 概要

`main` ブランチへの push をトリガーに GitHub Actions が自動デプロイを実行する。変更されたディレクトリに応じてワークフローが選択される。

### 6.2 トリガー条件

| ワークフロー | ファイル | トリガーパス |
|---|---|---|
| Deploy Frontend | `deploy-frontend.yml` | `frontend/**`, `.github/workflows/deploy-frontend.yml` |
| Deploy Backend | `deploy-backend.yml` | `backend/**`, `.github/workflows/deploy-backend.yml` |

### 6.3 デプロイフロー

```
git push origin main
       │
       ▼
GitHub Actions 起動
       │
       ├─ AWS 認証 (OIDC → AssumeRole)
       │
       ├─ ECR ログイン
       │
       ├─ Docker ビルド & ECR push (タグ: git commit SHA)
       │
       ├─ ECS タスク定義を新イメージで更新
       │
       └─ ECS サービス ローリングアップデート (安定確認まで待機)
```

### 6.4 必要な GitHub Secrets

| シークレット名 | 内容 |
|---|---|
| `AWS_ROLE_ARN` | GitHub Actions が AssumeRole する IAM ロール ARN |

### 6.5 認証方式

GitHub Actions から AWS への認証は **OIDC (OpenID Connect)** を使用し、長期的な AWS アクセスキーを管理しない。許可対象は `yagishita-hironori/helloworld` リポジトリの `main` ブランチのみ。

## 7. ローカル開発環境

### 7.1 前提条件

| ツール | バージョン |
|---|---|
| .NET SDK | 8.0 以上 |
| Node.js | 20 以上 |
| Docker Desktop | 任意 |

### 7.2 起動手順

**バックエンド:**

```bash
cd backend
dotnet run
# http://localhost:5000 で起動
```

**フロントエンド:**

```bash
cd frontend
npm install
npm start
# http://localhost:5173 で起動 (Vite)
```

## 8. セキュリティ設計

| 項目 | 対策 |
|---|---|
| AWS 認証情報 | OIDC による一時トークン使用、長期キー不使用 |
| DB 接続文字列 | AWS Secrets Manager で管理、コードに埋め込まない |
| コンテナ通信 | ALB 経由、コンテナ間は VPC 内通信 |
| IAM | 最小権限原則に基づくロール設計 |
