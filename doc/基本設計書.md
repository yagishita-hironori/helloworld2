# 基本設計書

**システム名:** Hello World Web Application (Next.js版)
**作成日:** 2026-03-18
**対象リポジトリ:** https://github.com/yagishita-hironori/helloworld2

---

## 1. システム概要

Next.js をフロントエンド、ASP.NET Core をバックエンド API として、AWS ECS/Fargate 上で稼働する Web アプリケーションのサンプル構成。ALB によるパスベースルーティング、RDS (SQL Server) によるデータ永続化、GitHub Actions による CI/CD を備える。

beans-web-test (React SPA / Nginx 構成) を Next.js SSR 構成に移行したものである。

---

## 2. システムアーキテクチャ

```
[ ブラウザ ]
     │ HTTP
     ▼
   ALB (Application Load Balancer)
     ├─ /           → frontend コンテナ (Next.js / Node.js)
     └─ /api/*      → backend コンテナ (ASP.NET Core API)
                                            ↓
                                    RDS (SQL Server)
```

### 構成要素

| コンポーネント | 技術スタック | ホスティング |
|---|---|---|
| フロントエンド | Next.js 14 / React 18 / Node.js 20 | ECS Fargate |
| バックエンド API | ASP.NET Core (.NET 8.0) | ECS Fargate |
| データベース | SQL Server (RDS) | Amazon RDS |
| ロードバランサー | Application Load Balancer | AWS ALB |
| コンテナレジストリ | Amazon ECR | AWS ECR |
| ログ管理 | Amazon CloudWatch Logs | AWS CloudWatch |

## 3. フロントエンド設計

### 3.1 技術仕様

| 項目 | 内容 |
|---|---|
| フレームワーク | Next.js 14.2.5 |
| ライブラリ | React 18.3.1 |
| レンダリング方式 | SSR (サーバーサイドレンダリング) |
| ルーティング | App Router (ファイルベース) |
| Web サーバー | Node.js 20 (standalone モード) |
| コンテナポート | 3000 |
| ローカル開発ポート | 3000 |

### 3.2 ディレクトリ構成

```
frontend/
├─ Dockerfile
├─ next.config.js       # standalone 出力・API リライト設定
├─ package.json
└─ app/
   ├─ layout.jsx        # ルートレイアウト
   ├─ page.jsx          # トップページ（SSR で /api/hello を呼び出し）
   ├─ health/
   │  └─ route.js       # GET /health（ALB ヘルスチェック用）
   └─ api/
      └─ (バックエンドへ ALB 経由でルーティング)
```

### 3.3 API 連携

#### SSR 時（サーバーサイド）

`page.jsx` がサーバーサイドで直接バックエンドを呼び出す。

```
Next.js コンテナ (ECS)
  → BACKEND_URL 環境変数 (ALB の URL)
  → ALB の /api/* ルール
  → バックエンドコンテナ (ECS)
```

#### ブラウザからのリクエスト時

`next.config.js` の rewrites により `/api/*` をバックエンドに転送する。

```
ブラウザ → ALB → /api/* ルール → バックエンドコンテナ
```

### 3.4 next.config.js の設定

| 設定 | 内容 |
|---|---|
| `output: 'standalone'` | 最小限のファイルのみを含む standalone ビルドを生成。Node.js で直接起動可能 |
| `rewrites` | `/api/:path*` を `BACKEND_URL` に転送。ローカル開発時は `http://localhost:5000` にフォールバック |

### 3.5 ヘルスチェック

| パス | 実装 | 用途 |
|---|---|---|
| `/health` | `app/health/route.js` (Next.js Route Handler) | ALB フロントエンドターゲットグループのヘルスチェック |

> **設計上の注意:** `/api/health` は ALB のリスナールールでバックエンドに転送されるため、フロントエンドのヘルスチェックは `/health` に配置する。

### 3.6 Docker ビルド（マルチステージ）

| ステージ | ベースイメージ | 処理 |
|---|---|---|
| build | node:20-alpine | `npm ci` → `npm run build`（`.next/standalone` に出力） |
| runtime | node:20-alpine | standalone ディレクトリと static ファイルのみをコピー |

```dockerfile
# build stage
FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# runtime stage
FROM node:20-alpine AS runtime
WORKDIR /app
ENV NODE_ENV=production
COPY --from=build /app/.next/standalone ./
COPY --from=build /app/.next/static ./.next/static
EXPOSE 3000
ENV PORT=3000
ENV HOSTNAME=0.0.0.0
CMD ["node", "server.js"]
```

### 3.7 環境変数

| 変数名 | 本番の値 | ローカルのデフォルト | 用途 |
|---|---|---|---|
| `BACKEND_URL` | `http://helloworld-alb-407709868.ap-northeast-1.elb.amazonaws.com` | `http://localhost:5000` | SSR 時のバックエンド呼び出し先 |
| `NODE_ENV` | `production` | - | Node.js 動作モード |
| `PORT` | `3000` | - | リッスンポート |
| `HOSTNAME` | `0.0.0.0` | - | リッスンアドレス |

│      ap-northeast-1.elb.amazonaws.com       │

│ Listener Rule:                              │
│   Priority 20: /api/* → backend TG         │
│   Default:     /      → frontend TG        │

          │                        │
          │ :3000                  │ :8080
          ▼                        ▼
┌──────────────────┐    ┌──────────────────────┐
│ ECS Service      │    │ ECS Service          │

│ Next.js / Node   │    │ ASP.NET Core         │

                                   │

### 5.2 ECS タスク定義

| 項目 | フロントエンド | バックエンド |
|---|---|---|
| ファミリー名 | helloworld2-frontend | helloworld2-backend |
| CPU | 512 (0.5 vCPU) | 512 (0.5 vCPU) |
| メモリ | 1024 MB | 1024 MB |
| コンテナポート | 3000 | 8080 |
| ネットワークモード | awsvpc | awsvpc |
| 起動タイプ | FARGATE | FARGATE |
| CloudWatch ロググループ | /ecs/helloworld2-frontend | /ecs/helloworld2-backend |

> **フロントエンドのメモリが 1024 MB の理由:** Next.js の SSR は Node.js プロセスで動作するため、React SPA を Nginx で配信していた元構成（512 MB）より多くのメモリが必要。

### 5.3 ALB ルーティング

| Priority | 条件 | 転送先 |
|---|---|---|
| 20 | パス: `/api/*` | helloworld2-backend-tg（ポート 8080） |
| デフォルト | - | helloworld2-frontend-tg（ポート 3000） |

## 6. CI/CD 設計

### 6.1 概要

`main` ブランチへの push をトリガーに GitHub Actions が自動デプロイを実行する。変更されたディレクトリに応じてワークフローが選択される。

### 6.2 トリガー条件

| ワークフロー | ファイル | トリガーパス |
|---|---|---|
| Deploy Frontend | `deploy-frontend.yml` | `frontend/**`, `.github/workflows/deploy-frontend.yml` |
| Deploy Backend | `deploy-backend.yml` | `backend/**`, `.github/workflows/deploy-backend.yml` |

### 6.3 デプロイフロー

```
git push origin main
       │
       ▼
GitHub Actions 起動
       │
       ├─ AWS 認証（OIDC → AssumeRole）
       │
       ├─ ECR ログイン
       │
       ├─ Docker ビルド & ECR push（タグ: git commit SHA）
       │
       ├─ ECS タスク定義を新イメージで更新
       │
       └─ ECS サービス ローリングアップデート（安定確認まで待機）
```

### 6.4 必要な GitHub Secrets

| シークレット名 | 内容 |
|---|---|
| `AWS_ROLE_ARN` | `arn:aws:iam::869935101124:role/github-actions-helloworld2-deploy` |

### 6.5 認証方式

GitHub Actions から AWS への認証は **OIDC (OpenID Connect)** を使用し、長期的な AWS アクセスキーを管理しない。許可対象は `yagishita-hironori/helloworld2` リポジトリの `main` ブランチのみ。

## 7. 通信の流れ

### 7.1 画面表示（SSR）

```
1. ブラウザが ALB にアクセス
2. ALB がデフォルトルールで frontend TG に転送
3. Next.js がサーバーサイドで BACKEND_URL 経由で /api/hello を呼び出す
4. Next.js が HTML をレンダリングしてブラウザに返す
5. ブラウザが完成した HTML を表示
```

### 7.2 ブラウザからの API 呼び出し

```
1. ブラウザが /api/* にリクエスト
2. ALB が Priority 20 のルールで backend TG に転送
3. ASP.NET Core API が処理
4. JSON を返す
```

## 9. セキュリティ設計

| 項目 | 対策 |
|---|---|
| AWS 認証情報 | OIDC による一時トークン使用、長期キー不使用 |
| DB 接続文字列 | AWS Secrets Manager で管理、コードに埋め込まない |
| コンテナ間通信 | ALB 経由、VPC 内通信 |
| IAM | 最小権限原則に基づくロール設計 |
| ECS タスク | Private Subnet への移行を将来的に検討（現在は Public Subnet） |
